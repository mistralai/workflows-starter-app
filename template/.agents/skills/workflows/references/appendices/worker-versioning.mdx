# Worker Versioning

Worker versioning enables safe deployments of workflow code changes by ensuring that running workflows continue executing on the same code version they started with, while new workflows are routed to the latest version.

## Overview

Mistral Workflows implements Temporal's Worker Deployment Versioning with **PINNED** behavior by default. This means:

- **Workflows stay on their starting version**: Once a workflow starts on a specific build, it remains on that build for its entire lifetime
- **New workflows use the current version**: Newly started workflows are automatically routed to the most recent build
- **Zero-downtime deployments**: Multiple worker versions can run simultaneously during rollouts

## When to Use Worker Versioning

### Production Deployments

Worker versioning is **essential** for production environments where:

- Workflows can run for hours, days, or weeks
- You need to deploy code changes without disrupting running workflows
- You want to gradually roll out new versions with canary deployments
- Multiple versions of your application may be running simultaneously

### Local Development

Worker versioning is **optional** for local development:

- Disabled by default for simplicity
- Can be enabled for testing versioning behavior
- Auto-registration handles version setup automatically

## Architecture

### Key Components

1. **Worker Deployment**: A logical grouping of workers (e.g., "payment-processor")
2. **Build ID**: A unique identifier for a specific code version (e.g., "v1.2.3-abc123")
3. **Current Version**: The build ID that new workflows are routed to
4. **Versioning Behavior**: PINNED (workflows stay on their starting build)

### How It Works

```
┌─────────────────────────────────────────────────────────────┐
│                      Temporal Server                        │
│                                                             │
│  Deployment: "payment-processor"                            │
│  ├─ Current Version: v2.0.0                                 │
│  ├─ Build v1.0.0 (2 running workflows)                      │
│  └─ Build v2.0.0 (5 running workflows)                      │
└────────────────────────────┬───-──────-─────────────────────┘
                             │
                ┌────────────┴────────────┐
                │                         │
        ┌───────▼────────┐         ┌──────▼─────────┐
        │  Worker v1.0.0 │         │  Worker v2.0.0 │
        │                │         │                │
        │  Handles only  │         │  Handles only  │
        │  workflows     │         │  workflows     │
        │  pinned to     │         │  pinned to     │
        │  v1.0.0        │         │  v2.0.0        │
        └────────────────┘         └────────────────┘
```

## Configuration

### Environment Variables

Worker versioning can be configured using two sets of environment variables:

#### Temporal Worker Controller (Production)

When using the [Temporal Worker Controller](https://docs.temporal.io/production-deployment/worker-controller) operator in Kubernetes:

```bash
# Set automatically by the controller
TEMPORAL_DEPLOYMENT_NAME=mistral-abraxas/payment-processor
TEMPORAL_WORKER_BUILD_ID=v1.2.3-abc123

# Versioning is auto-enabled when these are present
# Controller manages version promotion (no auto-registration)
```

#### Manual Configuration (Local/Testing)

For local development or non-controller deployments:

```bash
# Manual configuration
DEPLOYMENT_NAME=payment-processor
BUILD_ID=v1.2.3

# Worker auto-registers as current version (unless overridden below)
```

### Configuration Reference

| Variable                          | Description                                | Default |
| --------------------------------- | ------------------------------------------ | ------- |
| `DEPLOYMENT_NAME`                 | Deployment name (manual)                   | `None`  |
| `BUILD_ID`                        | Build ID (manual)                          | `None`  |
| `TEMPORAL_DEPLOYMENT_NAME`        | Deployment name (controller)               | `None`  |
| `TEMPORAL_WORKER_BUILD_ID`        | Build ID (controller)                      | `None`  |
| `WORKER_AUTO_REGISTER_AS_CURRENT` | Auto-register override (`true` by default) | `true`  |

### Auto-Registration Logic

Mistral Workflows automatically determines whether to register the worker as the current version:

- Controller mode: `WORKER_AUTO_REGISTER_AS_CURRENT` is ignored and auto-registration is always disabled.
- Manual mode: auto-registration defaults to `true`, but can be disabled with `WORKER_AUTO_REGISTER_AS_CURRENT=false`.

**Scenarios:**

| Scenario              | `TEMPORAL_DEPLOYMENT_NAME` | `DEPLOYMENT_NAME`                                | Auto-Register?             |
| --------------------- | -------------------------- | ------------------------------------------------ | -------------------------- |
| **Controller (Prod)** | ✅ Set                     | ❌                                               | ❌ No (controller manages) |
| **Manual (Local)**    | ❌                         | ✅ Set                                           | ✅ Yes (auto-register)     |
| **Explicit Disable**  | ❌                         | ✅ Set + `WORKER_AUTO_REGISTER_AS_CURRENT=false` | ❌ No                      |

## Deployment Flow

### How Worker Updates Work

When you deploy a new worker version, the Temporal Worker Controller orchestrates a zero-downtime rollout:

```
┌─────────────────────────────────────────────────────────────────┐
│ Step 1: Deploy New Version                                      │
│ ─────────────────────────────                                   │
│ $ helm upgrade ... --set workflows-worker.defaults.image.tag=v2 │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 2: Controller Creates New Deployment                       │
│ ────────────────────────────────────────                        │
│ • Generates unique build ID: v2.0.0-abc123                      │
│ • Creates k8s Deployment with new pods                          │
│ • Sets TEMPORAL_WORKER_BUILD_ID env var                         │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 3: New Workers Register with Temporal                      │
│ ─────────────────────────────────────────────                   │
│ • Pods start and connect to Temporal                            │
│ • Register with build ID v2.0.0-abc123                          │
│ • Wait for controller to promote                                │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 4: Controller Promotes New Version (Based on Strategy)     │
│ ──────────────────────────────────────────────────────────────  │
│                                                                 │
│ AllAtOnce Strategy:                                             │
│   → Immediately sets v2.0.0-abc123 as current (100% traffic)    │
│                                                                 │
│ Progressive Strategy:                                           │
│   → Ramp 1: 20% traffic to v2 (pause 10m)                       │
│   → Ramp 2: 90% traffic to v2 (pause 5m)                        │
│   → Ramp 3: 100% traffic to v2                                  │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 5: Workflow Routing (PINNED Behavior)                      │
│ ─────────────────────────────────────────────                   │
│ • New workflows → v2.0.0-abc123 workers                         │
│ • Running workflows (started on v1) → stay on v1.0.0-xyz        │
│ • No workflows switch versions mid-execution                    │
└─────────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 6: Old Version Cleanup                                     │
│ ────────────────────────────                                    │
│ • v1 workers keep running while v1 workflows execute            │
│ • When last v1 workflow completes → status: "Drained"           │
│ • After sunset.scaledownDelay → scale to 0 replicas             │
│ • After sunset.deleteDelay → delete deployment                  │
└─────────────────────────────────────────────────────────────────┘
```

### Production Deployment Example

**Helm values.yaml:**

```yaml
workflows-worker:
  temporal:
    namespace: production
    hostPort: temporal-frontend:7233
  defaults:
    image:
      tag: v1.2.3 # ← Update this to trigger rollout
  workers:
    - name: payment-processor
      taskQueue: payments
      replicas: 3
      rollout:
        strategy: Progressive # Gradual rollout
        steps:
          - rampPercentage: 20
            pauseDuration: 10m
          - rampPercentage: 90
            pauseDuration: 5m
      sunset:
        scaledownDelay: 30m # Wait 30min after drained
        deleteDelay: 2h # Delete after 2h
```

**Deployment commands:**

```bash
# 1. Update image tag to new version

# 2. Deploy
helm upgrade mistral-abraxas . -f values.yaml

# 3. Monitor rollout
kubectl get temporalworkerdeployment -n mistral-abraxas -w

# 4. Watch pods (you'll see both old and new versions)
kubectl get pods -n mistral-abraxas | grep payment-processor
# payment-processor-v1-2-3-abc   1/1  Running  (handling old workflows)
# payment-processor-v1-2-4-xyz   1/1  Running  (handling new workflows)
```

**Timeline:**

- **t=0**: Deploy v1.2.4
- **t=0-30s**: New pods start, register with Temporal
- **t=30s**: Controller promotes to 20% → 20% of new workflows use v1.2.4
- **t=10m30s**: Controller promotes to 90% → 90% of new workflows use v1.2.4
- **t=15m30s**: Controller promotes to 100% → all new workflows use v1.2.4
- **t=15m30s+**: Old v1.2.3 workflows complete, pods drain
- **t=45m30s**: Last v1.2.3 workflow done → scale v1.2.3 to 0 replicas
- **t=2h15m30s**: Delete v1.2.3 deployment

**No code changes needed!** The controller handles everything.

### Local Development

**Enable versioning locally:**

```bash
# .env.local
DEPLOYMENT_NAME=local-payment-processor
BUILD_ID=dev-$(git rev-parse --short HEAD)
```

**Run worker:**

```bash
uv run python -m examples.payment_workflow
```

**What happens:**

1. Worker starts with versioning enabled
2. After ~2.5 seconds, worker auto-registers as current version
3. All workflows route to this worker
4. No manual registration needed!

## PINNED Behavior Explained

Mistral Workflows uses **PINNED** versioning behavior by default, which provides the strongest guarantees for workflow consistency:

### What PINNED Means

```
Workflow Lifecycle with PINNED Behavior:
──────────────────────────────────────────

Workflow Start (v1.0.0)
    ↓
    ├─→ Activity execution → v1.0.0 worker
    ├─→ Child workflow → v1.0.0 worker
    ├─→ Continue-as-new → v1.0.0 worker
    ├─→ Workflow sleeps for 30 days...
    └─→ Workflow resumes → STILL v1.0.0 worker

Even if v2.0.0, v3.0.0, v4.0.0 are deployed,
this workflow STAYS on v1.0.0 for its entire life.
```

### Why PINNED?

**Guarantees:**

- ✅ Workflow code never changes mid-execution
- ✅ Activities always use the same code version
- ✅ No compatibility issues between workflow versions
- ✅ Safe to make breaking changes in new versions

**Use cases:**

- Long-running workflows (hours, days, weeks)
- Workflows with complex state machines
- Production deployments requiring high reliability
- When workflow code changes frequently

### Real-World Example

```python
# v1.0.0 - Initial deployment
@workflow.defn
class PaymentWorkflow:
    @workflow.run
    async def run(self, amount: int) -> str:
        await workflow.execute_activity(
            charge_card, amount,
            start_to_close_timeout=timedelta(hours=1)
        )
        return "paid"
```

**Day 1**: Start 100 payment workflows on v1.0.0

```python
# v2.0.0 - Breaking change: new required parameter
@workflow.defn
class PaymentWorkflow:
    @workflow.run
    async def run(self, amount: int, currency: str) -> str:  # ← New param
        await workflow.execute_activity(
            charge_card, amount, currency,  # ← Updated call
            start_to_close_timeout=timedelta(hours=1)
        )
        return f"paid {amount} {currency}"
```

**Day 2**: Deploy v2.0.0 with breaking change

**Result:**

- ✅ Old workflows (v1.0.0): Continue running on v1.0.0 workers
- ✅ New workflows (v2.0.0): Start on v2.0.0 workers with `currency` param
- ✅ No crashes or incompatibilities!

Without PINNED versioning, old workflows would try to run on v2.0.0 workers and fail due to missing `currency` parameter.

## Deployment Strategies

### All-At-Once Deployment

Immediately routes 100% of new workflows to the new version:

```yaml
rollout:
  strategy: AllAtOnce
```

**Use cases:**

- Non-production environments
- Low-risk changes
- Immediate cutover needed

### Progressive Deployment (Recommended)

Gradually increases traffic to the new version:

```yaml
rollout:
  strategy: Progressive
  steps:
    - rampPercentage: 10
      pauseDuration: 5m
    - rampPercentage: 50
      pauseDuration: 10m
    - rampPercentage: 100
      pauseDuration: 0s
```

**Use cases:**

- Production deployments
- Canary testing
- Risk mitigation

### Manual Deployment

Requires explicit promotion:

```yaml
rollout:
  strategy: Manual
```

### Cleanup

**Old versions are automatically cleaned up:**

- When all pinned workflows complete (Drained state)
- Based on `sunset.scaledownDelay` configuration
- Controller handles pod termination

```yaml
sunset:
  scaledownDelay: 30m # Wait 30min after drain
  deleteDelay: 2h # Delete deployment after 2h
```

## Troubleshooting

### Worker Not Registering as Current

**Symptoms:**

- Workflows not routing to new worker
- Log shows: "Waiting for worker deployment to be available"

**Solutions:**

1. Wait up to 2 minutes for auto-registration retries
2. Check worker is polling for tasks: `temporal deployment describe`
3. Verify deployment name is consistent across workers
4. Check Temporal server connectivity

### Workflows Stuck in RUNNING

**Symptoms:**

- Workflow starts but never completes
- No worker appears to be handling it

**Solutions:**

1. Verify current version is set: `temporal deployment describe`
2. Check worker with correct build_id is running
3. Ensure namespace matches between worker and workflow
4. Check worker logs for errors

### Version Conflicts

**Symptoms:**

- Error: "Namespace already used by another worker"

**Solutions:**

1. For tests: Set `ALLOW_OVERRIDE_NAMESPACE=true`
2. For production: Use unique namespace per environment
3. Check no other workers are using the same namespace

### Manual Registration Needed

**Symptoms:**

- Controller environment but auto-registration happens
- Don't want auto-registration

**Solutions:**
Set explicit disable:

```bash
WORKER_AUTO_REGISTER_AS_CURRENT=false
```
