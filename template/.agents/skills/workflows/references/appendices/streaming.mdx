# Streaming Architecture

This document covers the technical implementation details of Workflows Events Streaming. For a developer-focused guide on how to use streaming in your workflows, see the [Streaming Guide](../guides/streaming.mdx).

## Overview

Workflows Events Streaming enables real-time Server-Sent Events (SSE) from workflows and activities using [NATS JetStream](https://docs.nats.io/nats-concepts/jetstream) as the underlying message broker.

### Key Characteristics

- **Persistence**: Messages are stored in JetStream with configurable TTL (default: 15 minutes)
- **Replay**: Events can be replayed from any point in the stream history
- **Isolation**: Events are scoped per namespace, workflow, and activity
- **Size Limit**: NATS imposes a 1MB payload limit per message

## NATS Subject Hierarchy

The subject hierarchy enables precise filtering and routing of events. Every event is published to a subject that encodes its full context.

### Subject Format

```
{stream_prefix}_{namespace}.{workflow_name}.{parent_exec_id}.{exec_id}.{topic}.{scope}
```

| Segment          | Description                                          |
| ---------------- | ---------------------------------------------------- |
| `stream_prefix`  | Always `workflows_stream`                            |
| `namespace`      | Customer/workspace identifier (sanitized)            |
| `workflow_name`  | Name of the workflow                                 |
| `parent_exec_id` | Parent workflow's execution ID (for sub-workflows)   |
| `exec_id`        | Current workflow's execution ID                      |
| `topic`          | User-defined stream name (e.g., `token`, `progress`) |
| `scope`          | Either `workflow` or activity identifiers            |

### Scope Suffixes

**Workflow events** end with the literal suffix `.workflow`:

```
workflows_stream_ns.my-workflow.abc123.abc123.status.workflow
```

**Activity events** include the `activity` scope prefix and metadata:

```
workflows_stream_ns.my-workflow.abc123.abc123.token.activity.my_activity.1.1
                                                  в””в”Җ activity.{name}.{id}.{attempt}
```

### Parent/Child Workflow Relationship

For **top-level workflows**, `parent_exec_id` equals `exec_id` (duplicated):

```
workflows_stream_ns.parent-wf.exec123.exec123.status.workflow
                            в””в”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ same ID
```

For **child workflows**, `parent_exec_id` references the parent:

```
workflows_stream_ns.child-wf.parent123.child456.status.workflow
                           в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳв””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                           parent ID  child ID
```

This enables:

- Filtering all events for a workflow tree by `parent_exec_id`
- Distinguishing parent vs child workflow events
- Tracing lineage in complex workflow hierarchies

## Internal Implementation

### Interceptor Architecture

Streaming context is managed by two interceptors:

1. **ContextHandlerInterceptor** (always enabled): Creates fresh `WorkflowContext` from `workflow.info()` for every workflow execution. Ensures correct `parent_workflow_exec_id` is set based on Temporal's parent relationship. Also handles `PayloadWithContext` wrapping/unwrapping for cross-worker context propagation.

2. **WorkflowStreamingInterceptor** (always enabled): Publishes internal status events (`WORKFLOW_RUNNING`, `ACTIVITY_RUNNING`, etc.) at workflow and activity boundaries.

### Context Propagation

The `WorkflowContext` contains:

```python
class WorkflowContext:
    namespace: str
    execution_id: str  # Current workflow's execution ID
    parent_workflow_exec_id: str | None  # Parent's execution ID (if sub-workflow)
```

This context is:

- Created fresh for each workflow from `workflow.info()`
- Stored in a contextvar for activities to access
- Used to construct NATS subjects with correct lineage

### Stream Events

All events conform to the `StreamEvent` schema:

```python
class StreamEvent:
    stream: str  # Topic name (e.g., "token")
    sequence: int  # Monotonic sequence within the workflow
    timestamp_unix_nano: int
    data: Any  # User payload (must be serializable)
    nats_context: StreamEventNatsContext
    workflow_context: StreamEventWorkflowContext
    activity_context: StreamEventWriteActivityContext | None
    metadata: dict
```

## Configuration

### Environment Variables

```bash
# Required
NATS_ENABLED=true
NATS_HOST="nats://localhost:4222"

# Timeouts & Retries
NATS_READ_TIMEOUT=5.0
NATS_IDLE_HEARTBEAT=2.0
NATS_CONSUMER_INACTIVITY_TIMEOUT=300.0
NATS_CONNECTION_RETRIES=5
NATS_CONNECTION_DELAY=2.0

# Stream Settings
NATS_MESSAGE_MAX_AGE=900.0        # 15 minutes TTL
NATS_CONSUMER_BATCH_SIZE=5
NATS_QUEUE_MAX_SIZE=10000         # Bounded queue to prevent OOM
```

### Stream Lifecycle

1. **Stream Creation**: One JetStream stream per namespace, created on first publish
2. **Subject Binding**: Stream accepts all subjects matching `worflows_stream_{namespace}.>`
3. **Message Retention**: Messages expire after `NATS_MESSAGE_MAX_AGE` seconds
4. **Consumer Cleanup**: Inactive consumers are removed after `NATS_CONSUMER_INACTIVITY_TIMEOUT`

## Filtering with NATS Wildcards

NATS wildcards enable flexible subscription patterns:

| Pattern                    | Matches                                      |
| -------------------------- | -------------------------------------------- |
| `*.*.*.*.token.>`          | All token events (workflow + activity)       |
| `*.*.*.*.*.activity.>`     | All activity events across all workflows     |
| `*.my-workflow.*.*.*.>`    | All events from `my-workflow`                |
| `*.*.parent123.*.*.>`      | All events in a workflow tree (by parent ID) |
| `*.*.*.exec456.*.workflow` | Workflow-level events for specific execution |
